# 纯 HTML 2048 游戏 Requirements

## 1. Overview
本功能目标是实现一个纯 HTML/CSS/JavaScript 的 2048 单页游戏，支持键盘与触屏操作，可离线运行，不依赖后端服务。

## 2. Goals and Scope
### 2.1 Goals
- 提供完整可玩的 4x4 2048 游戏体验。
- 提供清晰反馈：当前分数、最高分、胜利/失败状态。
- 保持代码结构清晰，便于后续扩展主题、棋盘尺寸和新规则。

### 2.2 In Scope
- 单页界面布局与样式。
- 核心游戏逻辑（生成、移动、合并、计分、结束判定）。
- 键盘方向键与移动端滑动支持。
- 本地存储最高分。
- 新游戏重开与结果提示。

### 2.3 Out of Scope
- 账号系统、排行榜、联机对战。
- 后端 API 与数据库。
- 复杂动画引擎和音效系统。

## 3. Personas and User Stories
### Personas
- 休闲玩家：希望快速打开页面即可开始游戏。
- 前端开发者：希望代码简单、可维护、易扩展。

### User Stories
1. 作为玩家，我希望按方向键或滑动即可移动方块，以便快速进行游戏。
2. 作为玩家，我希望看到当前分数和历史最高分，以便了解表现。
3. 作为玩家，我希望在到达 2048 或无法移动时收到提示，以便决定继续或重开。

## 4. Functional Requirements
### FR-001 初始化与开局
- When 页面首次加载时，系统应创建 4x4 棋盘并随机生成 2 个初始数字块（2 或 4）。
- If 用户点击“新游戏”，系统应重置棋盘、分数和状态，并重新生成初始数字块。

### FR-002 移动与合并规则
- When 用户输入上/下/左/右方向时，系统应按该方向压缩并合并数字块。
- If 同一行/列出现相邻且数值相同的方块，系统应在一次移动中仅合并一次并生成双倍值。
- While 本次移动没有导致棋盘变化时，系统应不生成新数字块。

### FR-003 回合推进
- When 一次有效移动导致棋盘变化时，系统应在空位随机生成 1 个新数字块（默认 90% 为 2，10% 为 4）。

### FR-004 分数与最高分
- When 方块发生合并时，系统应将新方块值累加到当前分数。
- If 当前分数超过历史最高分，系统应更新最高分并持久化到 localStorage。

### FR-005 结束状态与提示
- When 任意方块达到 2048 时，系统应提示“达成 2048”，并允许继续游戏。
- If 棋盘已满且无任何可合并相邻方块，系统应判定失败并提示“游戏结束”。

### FR-006 多端输入支持
- When 桌面端按方向键时，系统应触发对应移动。
- When 移动端发生有效滑动手势时，系统应触发对应移动。

## 5. Non-Functional Requirements
### NFR-001 性能
- 在主流桌面浏览器中，单次移动与重绘应在可感知流畅范围内（目标 < 16ms 的逻辑处理，渲染不明显卡顿）。

### NFR-002 兼容性
- 至少支持近两年主流 Chromium 内核浏览器与 Safari 移动端浏览器。

### NFR-003 可维护性
- 游戏状态、渲染逻辑、输入处理应分层组织，避免全部堆叠在单个超长函数。

### NFR-004 可用性
- 提供清晰的按钮、状态提示和基础视觉层次，移动端布局可正常操作。

### NFR-005 可观测性
- 关键状态变更（初始化、移动、结束判定）应有可调试的日志开关或可追踪代码路径。

## 6. Data Requirements
- GameState: board(4x4 number matrix), score(number), bestScore(number), gameOver(boolean), won(boolean), keepPlaying(boolean)
- UIState: message(提示文案), controls(新游戏/继续按钮状态)

## 7. Acceptance Criteria
- 用户可完整进行 2048 游戏流程：开始、移动、合并、计分、结束、重开。
- 键盘与触屏输入均可驱动游戏，且不会出现方向错乱。
- 刷新页面后最高分可保留，当前局面不要求持久化。
- 达成 2048 与失败均有可见提示。

## 8. Dependencies and Risks
### Dependencies
- 仅依赖浏览器原生能力（DOM、CSS、localStorage、TouchEvent/PointerEvent）。

### Risks
- 风险：触屏手势与页面滚动冲突。
  - 缓解：为游戏容器设置合适 touch-action，并限制最小滑动阈值。
- 风险：合并规则实现错误导致“重复合并”bug。
  - 缓解：为每个移动方向编写独立测试用例和边界用例。

## 9. Open Questions
- 是否需要“撤销一步（Undo）”功能（当前默认不做）。
- 是否需要支持可配置棋盘尺寸（当前固定 4x4）。
